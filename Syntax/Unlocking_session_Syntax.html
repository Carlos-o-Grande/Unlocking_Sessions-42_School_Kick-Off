The Byte & Types in C

~7 min
Slide 04 Live Demo — IDE

Nibbles & hex — 0x42 is not 42

S04

A nibble is 4 bits. Two nibbles make a byte. Each nibble maps to exactly one hex digit.

hex literal: 0x42
0100  0010 upper nibble  lower nibble
4 × 16 + 2 × 1 = 66 decimal 
#include <stdio.h>

int     main(void)
{
    int     n;
    int     upper;
    int     lower;

    n = 0x42;                   // hex literal — decimal value is 66, not 42!
    upper = (n >> 4) & 0x0F;    // shift right 4, mask — upper nibble
    lower = n & 0x0F;            // mask lower 4 bits — lower nibble
    printf("hex:          0x%02X\n", n);      // 0x42
    printf("decimal:      %d\n", n);           // 66
    printf("upper nibble: %d\n", upper);        // 4
    printf("lower nibble: %d\n", lower);        // 2
    return (0);
}
Hex is a compact way to write binary — one hex digit per nibble. This is why memory addresses and bitmasks are always shown in hex.
Slide 05 Table

Types — the compiler's memory contract

S05

A type tells the compiler: how many bytes to read and how to interpret them. That is all a type is.

Type Size (64-bit) Range / Notes
char 1 byte −128 → 127  |  unsigned: 0 → 255  |  holds one character
int 4 bytes −2,147,483,648 → 2,147,483,647
long 8 bytes ~±9.2 × 10¹⁸
float 4 bytes ~7 significant decimal digits
double 8 bytes ~15 significant decimal digits
size_t 8 bytes Unsigned · safe for sizes & indices · platform-adapted 
int    a = 7;
int    b = 2;
float  result;

result = a / b;           // int / int = int — result is 3.0, not 3.5
result = (float)a / b;    // explicit cast — result is 3.5
int ÷ int = int. Always. The decimal is silently discarded. One of the most common silent bugs in C.

On the nibble demo (S04): "The school is called 42. But if you write 0x42 in C — the hex literal — what does it equal?" Pause. Show the nibble breakdown. Upper nibble 4, lower nibble 2, total 4×16+2 = 66. Compile and run to confirm. This is a memorable moment that connects the school name to a real binary concept.

The >> shift and & 0x0F mask are bitwise operations — you don't need to teach them, just show the pattern and say "this extracts the upper or lower 4 bits".

"Why do you think memory addresses are always shown in hex rather than decimal?" — each byte = exactly 2 hex digits. Much more compact for raw memory values.

On types (S05): The core idea: types are a contract with the compiler. Without them, the compiler has no idea how many bytes to read at a given address.

On the implicit conversion demo: Type it live. The result being 3.0 surprises people every single time. The fix — explicit cast — is the lesson.

Likely Follow-up

"Is int always 4 bytes?"

On all 42 school machines (64-bit Linux), yes. The C standard only guarantees a minimum. That's why stdint.h provides int32_t and uint64_t for guaranteed-size needs.

Types control what the compiler reads. Flags control what the compiler tells you when something looks wrong.

Compiler Flags & Makefile

~5 min
Slide 06 Table

-Wall -Wextra -Werror — your first line of defence

S06

These three flags are mandatory at 42. Together they turn suspicious code into compile errors.

Flag What it catches
-Wall Unused variables · missing return · implicit function declarations
Assignment in condition if (x = 5) · integer overflow in constants
— the standard set of "probably a bug" warnings
-Wextra Signed/unsigned comparison · missing parameter names · unused parameters
Precedence warnings a & b == 0 · shadow variable warnings
— goes beyond -Wall to catch subtler issues
-Werror Treats every warning as a compile error.
Code does not compile if it has any warning. No exceptions.
The flags don't make your code better — they make the compiler louder when your code is suspicious. That loudness is a feature, not an obstacle.
Slide 07 Code

Makefile — the build contract

S07

Add source filenames to FILES to compile multi-file programs. This is the minimum viable Makefile.

CC     = cc
CFLAGS = -Wall -Wextra -Werror
NAME   = program
FILES  = main.c

all:
	$(CC) $(CFLAGS) $(FILES) -o $(NAME)

clean:
	rm -f $(NAME)

fclean: clean

re: fclean all

.PHONY: all clean fclean re
  • make — compile. make re — recompile from scratch
  • Add more sources to FILES separated by spaces — immediate multi-file support
  • fclean and clean both remove the binary — no object files in this setup
  • Extend with OBJS and pattern rules when your project grows
The CFLAGS are already set to the 42 mandatory flags. Your code compiles with the same strictness every time, on every machine.

Students sometimes disable warnings to make their code compile. This slide explains why that's backwards — warnings are telling you something. -Werror forces you to listen.

"Has anyone had a program compile successfully and then crash at runtime — with no warnings? What does that tell you about what warnings catch?" — most hands go up. Answer: warnings catch things that look suspicious at compile time, not everything that goes wrong at runtime.

On the Makefile (S07): Explain each variable. Show it working with the nibble demo code from S04 — make, then run the program. The simplicity is the point. Add a second .c file to FILES live to show it still works.

Likely Follow-up

"Can I silence a specific warning?"

Yes — with -Wno-warning-name. At 42, the norm flags are fixed and you shouldn't silence them. In real-world code you sometimes suppress known-intentional patterns. For now: if the compiler warns about it, fix it.

The flags will catch many boolean and assignment mistakes before they reach runtime. Let's look at those patterns now.

Slide 09 Code

Short-circuit evaluation & the = vs == trap

S09
  • && stops at the first false — right side never evaluates if left is false
  • || stops at the first true — right side never evaluates if left is true 
if (ptr->value == 42)          // crash if ptr is NULL — reads from address 0
    do_something();

if (ptr && ptr->value == 42)   // safe — right side only runs if ptr is not NULL
    do_something();

= vs == — the silent bug that compiles perfectly (-Wall will catch it):

int   x;

x = 0;
if (x = 5)    // assigns 5 to x, evaluates x (truthy) — always runs
    ...
if (x == 5)   // compares x to 5 — what you meant
    ...
Short-circuit is a pattern, not a trick. Always check a pointer is non-NULL before dereferencing it. -Wall warns on assignment inside a condition.

Classic traps — what does C actually read?

a & b == 0    // becomes: a & (b == 0)  — equality binds tighter than &
*p++          // becomes: *(p++)        — postfix ++ binds tighter than *
a = b == c    // becomes: a = (b == c) — comparison before assignment
!x + 1       // becomes: (!x) + 1     — ! is unary, binds very tight
Rule: when mixing operators from different rows in the table, add explicit parentheses. Parentheses are documentation — they show intent to the compiler and to the next reader.

On truth tables (S08): Before explaining, point at the orange-striped row (A=1, B=1). "Look at the same-coloured row across all four tables — when both inputs are 1, what does each operation give?" Let them read back.

"Look at the XOR table — when is XOR false? What do those rows have in common?" — only the rows where A equals B. XOR means "they're different." Useful in conditionals when you want exactly one of two conditions true.

On NAND and NOR (De Morgan section): "NAND is AND with the output flipped. NOR is OR with the output flipped. De Morgan's law tells us how to rewrite them — distribute the NOT and flip the operator. This comes up when you want to simplify a negated compound condition." Show the equivalence on paper if helpful.

On short-circuit (S09): The ptr example is the key. If ptr is NULL (0), the && short-circuits: right side never evaluates, no crash. This is why && and || are designed this way — it's not an accident.

"Walk me through what happens step by step when C evaluates if (x = 5)." — assign 5 to x, evaluate x (5 is nonzero = true), branch always taken. -Wall produces: warning: suggest parentheses around assignment used as truth value.

On the precedence table (S10): Don't read it aloud. Use it as visual reference. Walk through each trap example. On *p++: "postfix ++ binds before *, so the pointer advances — not the dereferenced value." They'll use this pattern in the defensive programming section.

Likely Follow-up

"Does C have a bool type?"

Yes — via #include <stdbool.h> in C99, giving bool, true, false. Most 42 code uses int as booleans. The logic is identical: 0 = false, nonzero = true.

Scope & Variable Lifetime

~4 min
Slide 11 Visual

Scope — { } is a life boundary

S11
int main(void) { int x = 10;                          BORN { int x = 99;                     BORN SHADOWS outer x printf("%d\n", x);   // prints 99 }                                    inner x DEAD printf("%d\n", x);    // prints 10 }
  • A variable lives from its declaration to the closing }
  • Shadowing: inner name hides outer — outer still exists, just unreachable
  • 42 norm: declare all variables at the top of the function — no nested declarations
The norm rule isn't arbitrary. Declarations at the top make the entire variable set visible at a glance.
Slide 12 Live Demo

Uninitialized variables — reading garbage

S12
int   x;
printf("%d\n", x);   // undefined behaviour — prints whatever was there before

int   x;
x = 0;               // explicit initialization — predictable

int   arr[5];
ft_memset(arr, 0, sizeof(arr));  // zero-initialize a whole array
  • An uninitialized local variable holds whatever memory happened to be there
  • C makes no guarantee — the value may differ on every run, every machine
  • Habit: initialize to 0 when you declare, even if you will assign later
Compile and run — show the garbage value. Run again. It may change. This is the behaviour named "undefined".

On scope (S11): Walk through line by line. "We enter the outer block. x=10 is born. Inner block opens. A new x=99 is born — it hides the outer. We see 99. Inner block closes — inner x dies. We're back to 10." Draw nested boxes if needed.

"If I declare a variable inside a while loop body — what happens to it on the next iteration?" In norm-compliant code this can't happen, but the concept of lifetime still applies at function scope.

On uninitialized vars (S12): Actually compile and run. Show the garbage. Run twice — it may differ. This visual demonstration is more memorable than any explanation. After: "Where exactly does that variable live in memory?" — let answers come. "Stack" is correct. "That's the next session."

Variables have scope and lifetime. Now: how do we use them in repetition?

Loops

~5 min
Slide 13 Code

While — the only loop you write at 42

S13
int   i;

i = 0;                       // 1. init      — before the loop
while (i < 5)                // 2. condition — tested before each iteration
{
    printf("%d\n", i);        // 3. body      — what you want to repeat
    i++;                      // 4. increment — move toward the exit condition
}                             // jumps back to 2
  • for exists and is the same machine — norm forbids it. Converting: step 1 goes before, step 4 goes to the body's end. Done.
  • Off-by-one: i < 5 → indices 0,1,2,3,4 (correct for a 5-element array). i <= 5 → 6 iterations, last is out of bounds.
  • Missing step 4 → infinite loop. The condition never changes.
  • Also forbidden by norm: for do…while
Every while loop has four parts. If one is missing — you have a bug or an infinite loop. Always ask: "What changes to eventually make the condition false?"
Slide 14 Live Demo

Nested loops — name your variables for the context

S14
int   row;
int   col;

row = 0;
while (row < 3)                // outer: controls which row we are on
{
    col = 0;
    while (col < 4)             // inner: runs completely for each row
    {
        printf("[%d,%d] ", row, col);
        col++;
    }
    printf("\n");
    row++;
}
  • Inner loop executes completely for each outer step: 3 × 4 = 12 total
  • row and col — not i and j. The name tells you which dimension you're in
  • In geometric code: x and y are better still — name what the value means
  • Classic bug: forgetting col = 0 inside the outer loop — demo this deliberately
When operating in two dimensions — grid, matrix, 2D space — your variable names should reflect those dimensions. i and j tell you nothing. row and col tell you everything.

On the while anatomy (S13): Number each part as you speak. Students know how to write loops — naming the four parts makes the structure explicit. "Which of the four parts is missing?" is a more useful debugging question than "what's wrong?"

On for: 20 seconds. "It exists. You'll see it everywhere. Norm forbids it. The conversion is one move — figure it out yourself."

"For a 5-element array — which is correct: i < 5 or i <= 5? Tell me the index of the last element first." Indices go 0 through 4. i < 5 is correct. i <= 5 reaches index 5 — outside the array.

On nested loops (S14): Build incrementally in the IDE. First outer loop printing just the row number. Then add inner loop. Then deliberately remove col = 0 inside the outer loop and run it. Show the bug. Restore. Students remember witnessed bugs.

On naming: make it natural, not a rule. "I could write i and j here — but look at the printf. If the outer variable is called row, I don't have to think about which loop is which while I'm reading."

Now that we can write loops correctly, let's talk about writing everything else correctly — starting with naming.

Clean Code — Writing Style

~4 min
Slide 15 Code

Names communicate intent

S15
// functions are verbs — they do something
int    count_words(char *sentence, char separator);
char   *find_next_token(char *input);
int    validate_input(char *buffer, size_t length);

// variables and parameters are nouns — they hold something
size_t  word_count;
char    *current_token;
int     is_valid;      // booleans: starts with is_ or has_

// names that carry dimension belong in the context
int     pixel_x;      // not i
int     pixel_y;      // not j
int     row;          // not i — even for simple loops
int     col;          // not j

// len is universal — always understood
size_t  len;
  • 42 norm: snake_case for everything, no camelCase
  • A name should answer: what does this hold or what does this do
  • If you need a comment to explain a variable name — rename the variable instead
The person reading your code at 2am debugging a segfault is you. Name things so that person can work without rereading every function.

This is about intention, not rules. The audience already writes code — this is about writing code that communicates.

"If a function is called process — what does it do?" They'll say "we don't know." Follow up: "What if it's called validate_email?" The contrast makes the principle obvious without a lecture.

The is_ / has_ prefix for booleans is a widely used convention. It makes conditions read like English: if (is_valid), while (has_more_tokens).

The "if you need a comment to explain the name — rename it" line usually gets a reaction. Let it land before moving on.

len is a universally understood abbreviation — there's no ambiguity. Don't spell it out as length_of_string; that's noise. Don't abbreviate further to l; that's a single letter with no meaning.

Good names prevent bugs. Defensive programming catches the bugs that names can't prevent.

Defensive Programming

~7 min
Slide 16 Code

Guard clauses — handle bad cases first

S16

Defensive programming: write code that handles what shouldn't happen — before it does.

Three levels of nesting
int process(char *s, int len)
{
    int   result;

    result = 0;
    if (s)
    {
        if (len > 0)
        {
            // work lives 3 levels deep
            result = 1;
        }
    }
    return (result);
}
Flat — one level
int process(char *s, int len)
{
    if (!s || len <= 0)
        return (0);
    // happy path — no nesting
    return (1);
}

Invert the bad conditions. Return early. Keep the main logic at the shallowest indentation level.

Every level of nesting is a place bugs hide. Flat code is readable code. Readable code is maintainable code.
Slide 17 Code

Magic numbers — only the compiler is magic

S17
✨ Only the compiler is magic.
You are a muggle. Name your constants.
#define your values — always.
❌ Magic (for compilers only)
char   buf[1024];
int    limit = 42;

if (count > 42)
    handle_overflow();

What is 1024? What is 42? Change one — did you find them all?

✓ Named constants
#define BUFFER_SIZE  1024
#define MAX_PLAYERS  42

char   buf[BUFFER_SIZE];
int    limit = MAX_PLAYERS;

if (count > MAX_PLAYERS)
    handle_overflow();

Change it in one place. Intent is visible. No hunting.

#define is compile-time: no memory cost, usable in array size declarations where variables are not. It is the C way to name things that never change.
Slide 18 Code

Header guards & const — systematic protection

S18
Header guard — prevent double inclusion
#ifndef MYPROJECT_H
# define MYPROJECT_H

# include <unistd.h>

int   my_function(void);

#endif

Without this: including the same header twice causes duplicate declarations → compile error.

const — making intent visible
const char  *s;
// data is immutable, pointer can change

char  *const s;
// pointer is immutable, data can change

size_t  ft_strlen(const char *str);
// promise: I will not modify your string
Header guards are the canonical C defensive pattern — every .h file you write should have one. const is a promise you make to the compiler and to whoever calls your function.
Slide 19 — Approach A Live Demo

Iterating without int i — approach A: sizeof

S19a
int     arr[] = {10, 20, 30, 40, 50};
size_t  len;
int     *ptr;

// sizeof(arr)    = 5 x 4 = 20 bytes (total array)
// sizeof(arr[0]) = 4 bytes     (one element)
// 20 / 4         = 5           (number of elements)
len = sizeof(arr) / sizeof(arr[0]);

ptr = arr;
while (ptr < arr + len)
{
    printf("%d\n", *ptr);
    ptr++;
}
  • The pointer is the iterator — no index variable needed
  • ptr++ advances by sizeof(int) bytes — it knows the type
  • ⚠️ This breaks if you pass arr to a function — sizeof returns the pointer size, not the array size. This has a name: pointer decay.
sizeof / sizeof works only when the array is declared in the same scope. Once passed to a function, the size information is gone. That mechanism is the next session's opening.
Slide 19 — Approach B Live Demo

Iterating without int i — approach B: pointer as iterator

S19b
// Works when the array has a sentinel — a value that marks the end
int   arr[] = {10, 20, 30, 40, 50, 0};   // 0 = end marker
int   *ptr;

ptr = arr;
while (*ptr)
{
    printf("%d\n", *ptr);
    ptr++;
}

// argv is already NULL-terminated — same pattern, no design needed
char  **arg;

arg = argv + 1;    // skip program name
while (*arg)
{
    printf("%s\n", *arg);
    arg++;
}
  • Strings already use this — '\0' is a sentinel. Strings are not special; they are char arrays with a contract
  • You need to design the contract: what value means "end"?
  • argv is a real example they have used — NULL-terminated array of char pointers
The sentinel approach requires a contract. Once you have one, the pointer drives itself — no index, no counter, no sizeof.

On guard clauses (S16): Show both side by side in the IDE. Ask: "Which would you rather debug at 2am?" The flat version always wins. The key move: invert the bad condition, return immediately, keep the work at the shallowest level.

"Give me a concrete scenario where not checking for NULL before using a pointer causes a problem — what exactly happens at runtime?" Guide toward: segmentation fault (SIGSEGV). You are writing to or reading from memory you do not own. The OS kills the process.

On the muggle slide (S17): Let the joke land before moving to the code. The "change it in one place" argument is the practical reason. Spend 30 seconds on the why, not the what.

On header guards (S18): Quick scenario: file A.c and B.c both include myheader.h, and A.c also includes B.c. Without guards, the header is processed twice — duplicate declarations, compile error. With the guard, the second include is a no-op. Every .h file you write should have one.

On const (S18): const left of * means data is const. const right of * means pointer is const. Function parameters marked const char * are a promise: "I will not modify your string." The compiler enforces that promise.

On sizeof approach (S19a): Walk through the math on screen. Then: "Now watch what happens when I move this to a function." Write a helper that receives int *arr and tries the same sizeof trick. Show it returns 2 (8/4 = 2). Wrong. "The array lost its size information. This is called pointer decay. That is the next session."

On sentinel approach (S19b): Connect to ft_strlen: they wrote while (str[i] != '\0') i++; — this is just while (*str) str++;. Same logic, pointer-driven. argv is a real example they have used.

Likely Follow-up

"What is the difference between NULL, '\0', and 0?"

In C all three compare equal — they are all zero. NULL is (void*)0 for pointers. '\0' is the null byte for string terminators. 0 is the integer zero. Using the right one communicates intent, even if the behaviour is identical.

You can write, defend, and iterate over C code. The last skill: reading a declaration you have never seen before.

Reading C — The cdecl Method

~4 min
Slide 20 Visual

The inside-out, right-then-left rule

S20

Start at the identifier. Go right first, then left, jump out of parentheses and repeat.

char  *(*f)(int, char *)

1
Start at f
2
Go right → hit ). Go left → *.  f is a pointer
3
Escape ( ). Go right → (int, char *).  to a function taking (int, char *)
4
Go left → char *.  returning char *
f is a pointer to a function taking (int, char *), returning char *
You won't write declarations like this often. But you will read them in headers, man pages, and other people's code. Being able to parse them calmly is a superpower.
Slide 21 Live Demo — cdecl.org

Practice — and validate at cdecl.org

S21
// Read these before I reveal — right-then-left from the name

int   (*arr)[5]
// arr is a pointer to array 5 of int

const char  *s
// s is a pointer to const char  (data immutable, pointer movable)

char  *const s
// s is a const pointer to char  (pointer immutable, data mutable)

Navigate live to cdecl.org — type a declaration, get plain English. Use it as validation, not as a crutch.

  • const left of * → data is const
  • const right of * → pointer is const
  • The position relative to the star is the rule
const left of * = data is const. const right of * = pointer is const.

Before showing the answer (S20): Put the declaration on screen and ask someone to attempt a reading. "Don't be afraid to be wrong — what do you think the outer star is doing?" The most common wrong answer: "it returns a pointer to char." The right framing: the parentheses around (*f) group the star with f, making f itself the pointer.

"Before I walk through it — tell me what you think the parentheses around (*f) are doing here. Why are they there?" Without them, char *f(int, char*) would be a function, not a pointer-to-function.

On cdecl.org (S21): Navigate live. Type the declarations in order. Let students read before you reveal. If internet is unavailable, the method stands on its own — skip the site.

Likely Follow-up

"When would I use a function pointer?"

Callbacks — passing a function as an argument. ft_lstmap takes a function pointer to apply to each list node. qsort() takes a comparator. Anywhere you want to parameterise behaviour without if/else chains. You will use this throughout your 42 projects.

Likely Follow-up

"Is there a rule for where to put const — before or after the type?"

const int * and int const * are identical. What matters is position relative to the star. Const left of star = data const. Const right of star = pointer const.
